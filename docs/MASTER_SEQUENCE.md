# MASTER_SEQUENCE: последовательность операций мастера

Документ описывает рекомендуемый алгоритм обмена с ESP32 slave по Modbus RTU.

## 1. Параметры линии
- Протокол: Modbus RTU
- UART: 19200, 8N1
- slave_id: из пусконаладки (NVS на slave)

## 2. Базовые правила
1. Формат большинства значений: `x10`.
2. Любой валидный запрос мастера = heartbeat для slave.
3. При отсутствии heartbeat >30 с slave уходит в AUTONOMOUS.
4. Возврат в REMOTE: 3 успешных цикла связи подряд.

## 3. Стартовая инициализация (после включения мастера)

### Шаг 1: Проверка связи
- Прочитать диагностику:
  - `128` MODE_STATE
  - `129` MODE_REASON
  - `130..131` LAST_MASTER_SEEN_MS
  - `132` GOOD_CYCLE_STREAK
  - `133` LAST_APPLY_STATUS

### Шаг 2: Установить порог радиации
- Записать `135` SOLAR_UPPER_THRESHOLD_X10 (например, `8000` = 800.0).

### Шаг 3: Передать/подтвердить расписание света
1. Записать `110..121` (staging schedule).
2. Записать `122` (`APPLY_CMD`) - каждая успешная запись считается попыткой apply.
3. Прочитать:
   - `125` APPLY_STATUS
   - `126..127` ACTIVE_CTRL_VERSION
4. Успех: `APPLY_STATUS=0` и изменение `ACTIVE_CTRL_VERSION`.
5. Ошибка: `APPLY_STATUS!=0`, версия не меняется.

## 4. Периодический runtime-цикл мастера

Рекомендуемый период: 1 сек (или быстрее при необходимости).

### Шаг 1: Передача радиации
- Записать в `134` SOLAR_RADIATION_X10 текущее значение солнечной радиации.
- Пока датчик не подключен к мастеру: писать `0`.

### Шаг 2: Чтение телеметрии
- Прочитать `0..8`:
  - AIR_TEMP, AIR_HUM, WATER_RAIL, WATER_GROW, WATER_UNDERTRAY,
    WATER_UPPER_HEAT, WINDOWS_POS_A, WINDOWS_POS_B, CURTAIN_POS

### Шаг 3: Чтение статусов режима
- Прочитать `128..133`.
- Дополнительно прочитать `136` LIGHT_REDUCTION_ACTIVE.

## 5. RTC-синхронизация (по токену)

Рекомендуется выполнять при reconnect и периодически (например, 1 раз в сутки).

1. Записать серверное время в:
   - `140` RTC_SET_HOUR (0..23)
   - `141` RTC_SET_MINUTE (0..59)
2. Записать новый ненулевой токен в `142` RTC_SET_TOKEN.
3. Прочитать `143` RTC_SET_APPLIED_TOKEN и `144` RTC_SET_RESULT.
4. Обработка считается завершенной, когда `143 == token`.

Идемпотентность:
- Повтор той же команды с тем же `token` не вызывает повторную установку RTC.

Интерпретация `144 RTC_SET_RESULT`:
- `2` APPLIED: RTC обновлен.
- `5` NOOP: расхождение ниже порога на slave (`RTC_SYNC_THRESHOLD_MIN`, сейчас 3 мин).
- `3` REJECT_RANGE: невалидные `hour/minute`.
- `4` FAILED: ошибка чтения/записи RTC.

## 6. Обновление расписания света на лету
Когда нужно поменять расписание:
1. Подготовить `110..121`.
2. Записать `122` (`APPLY_CMD`).
3. Проверить `125` и `126..127`.
4. Без записи в `122` active schedule не меняется.
5. Если команды apply приходят быстрее, чем slave успевает применить, действует очередь глубиной 1 (`latest wins`).

## 7. Интерпретация ключевых статусов
- `125 APPLY_STATUS`:
  - `0` APPLIED/OK
  - `2` INVALID_RANGE
  - `3` BUSY
  - `5` INTERNAL_ERROR
- `128 MODE_STATE`:
  - `0` REMOTE
  - `1` AUTONOMOUS
- `136 LIGHT_REDUCTION_ACTIVE`:
  - `0` Свет не урезан по радиации
  - `1` Свет принудительно 50% (радиация >= порога)

## 8. Логика света (для понимания мастером)
Slave сам управляет двумя реле света:
1. Вне активного расписания: оба реле OFF (0%).
2. В активном расписании и радиация ниже порога: оба реле ON (100%).
3. В активном расписании и радиация >= порога: одно реле OFF (50%).
4. Интервалы через полночь (`ON > OFF`) поддерживаются.

## 9. Минимальный набор запросов для надежной работы
Каждый цикл:
1. Write `134` (радиация)
2. Read `0..8` (телеметрия)
3. Read `128..136` (диагностика + свет)

Этого достаточно для heartbeat, мониторинга и управления светом через радиацию/расписание.
